# РОЗДІЛ 4. ПРОГРАМНА РЕАЛІЗАЦІЯ ТА ПРАКТИЧНА АПРОБАЦІЯ ПЛАТФОРМИ HPF-P

> **Примітка щодо визначення фармацевтичного портфеля.** У цьому розділі та у дисертаційному дослідженні загалом термін *фармацевтичний портфель* позначає **комерційний товарний портфель фармацевтичного виробничого підприємства** — сукупність SKU (товарних позицій), які компанія типу ПАТ «Фармак» або ПрАТ «Дарниця» виробляє, просуває та реалізує на ринку. Це *товарний* портфель, принципово відмінний від *клінічного* портфеля. Оптимізаційна задача: маючи N товарних позицій (SKU) діючого асортименту, визначити оптимальний розподіл ресурсів між M обраними позиціями для максимізації очікуваного прибутку за умов часткової спостережуваності.

---

## 4.1 Архітектура програмної платформи HPF-P

### 4.1.1 Загальна структура та компоненти системи

Платформа HPF-P (Holistic Portfolio Framework — Platform) реалізована як модульна серверно-клієнтська система, призначена для вирішення задачі оптимального розподілу ресурсів між товарними позиціями (SKU) фармацевтичного підприємства. Вибір архітектурного підходу визначається трьома ключовими вимогами: детермінованістю результатів (ідентичні вхідні дані завжди породжують ідентичний результат), прозорістю логіки прийняття рішень (повне трасування від вхідних даних до рекомендованих ваг), та здатністю адаптуватись до гетерогенної якості інформації, характерної для вітчизняного фармацевтичного ринку [1].

Система складається з трьох рівнів (Рисунок 4.1):

**Рівень 1 — Серверна частина (Backend).** Реалізована на базі фреймворку FastAPI (Python 3.12). Серверна частина виконує обчислювально інтенсивні операції: обробку та валідацію вхідних даних (Модуль M1), обчислення індексу DRI та рівня DRL (Модуль M2), ML-оцінювання (Модуль M3), багатостратегічну оптимізацію портфеля (Модуль M4), стохастичне симулювання Монте-Карло (Модуль M5) та часорядову симуляцію (Модуль M5b), а також збірку вихідних результатів (Модуль M6). Сервіс розгортається як systemd-служба `hpf-app` і прослуховує порт 8901.

**Рівень 2 — REST API.** Забезпечує єдину точку входу (`POST /api/analyze`) для клієнтів. Архітектура REST дозволяє відокремити обчислювальну логіку від клієнтського рівня, забезпечує горизонтальну масштабованість та спрощує інтеграцію з корпоративними ERP- та CRM-системами.

**Рівень 3 — Клієнтська частина (Frontend).** Включає веб-застосунок з попередньо визначеними сценаріями для п'яти українських фармацевтичних компаній, а також клієнтський рушій на JavaScript (сценарний режим) для інтерактивної демонстрації. Клієнтський рушій реалізує детерміновану генерацію синтетичних даних за допомогою алгоритму Mulberry32 PRNG із фіксованим початковим значенням (seed = 42 + idx×1000), що забезпечує відтворюваність результатів на різних сеансах [2].

Така архітектура відповідає вимогам, сформульованим у Розділі 2 щодо прозорості та аудитованості систем підтримки прийняття рішень у фармацевтичній галузі. Усі ключові рішення — від вибору стратегії оптимізації до призначення рівня DRL — фіксуються у вихідних даних і можуть бути верифіковані аналітиком.

Конвеєр HPF-P реалізує послідовну обробку з чітко визначеними інтерфейсами між модулями. Структуру конвеєра наведено у Таблиці 4.1.

| M2 | DRI/DRL діагностика | DataFrame + метадані | DRI (0–1), DRL (1–5), стратегія | Формула DRI = ΣwᵢRᵢ, порогова функція |

Особливістю архітектурного рішення є **детермінований вибір стратегії**: у разі якщо `deterministic=true`, глобальний PRNG ініціалізується фіксованим seed=42, що гарантує ідентичні результати для однакових вхідних даних. У стохастичному режимі (`deterministic=false`) базовий seed генерується випадково, однак кожна ітерація i використовує `seed(base + i)`, зберігаючи внутрішню відтворюваність ітерацій.

### 4.1.3 Інтерфейс прикладного програмування (REST API)

Платформа HPF-P надає RESTful API з єдиною основною точкою доступу. Специфікацію ендпоінтів наведено у Додатку Г.

Основний ендпоінт `POST /api/analyze` приймає JSON-запит із трьома полями:
- `csv_data` (string) — вміст CSV-файлу з часовими рядами продажів;
- `metadata` (object) — конфігураційний JSON із параметрами портфеля та SKU;
- `deterministic` (boolean, за замовчуванням `true`) — режим детермінізму PRNG.

Відповідь містить: `portfolio_weights` (рекомендовані ваги, DRI, DRL, CI по кожному SKU), `module_diagnostics` (діагностика по всіх модулях), `portfolio_metrics` (CVaR-95, очікувана дохідність, дисперсія), `simulation_results` (часорядові траєкторії та економічне порівняння) та `narrative` (автоматично генерований текстовий опис результатів).

Час обробки одного запиту для портфеля з 15–21 SKU (36 місяців історії, 1000 ітерацій MC, 500 шляхів TS-симуляції) складає 45–90 секунд на стандартному серверному обладнанні (4 vCPU, 8 GB RAM). Детальна специфікація схем запиту та відповіді наведена у Додатку Г.

Перевірка доступності сервісу: ендпоінт `GET /` повертає HTML-сторінку клієнтської частини зі статус-кодом 200. Стан служби контролюється systemd (`systemctl status hpf-app`), що забезпечує автоматичний перезапуск у разі збою.

### 4.1.4 Інтеграція з веб-середовищем

Платформа розгорнута у складі веб-системи з використанням Apache як зворотного проксі: запити за шляхом `/hpf/` перенаправляються на локальний порт 8901. Така конфігурація забезпечує безпечний зовнішній доступ без безпосереднього відкриття Python-сервера в мережу.

Клієнтська частина реалізована на чистому JavaScript без залежностей від важких фреймворків, що забезпечує мінімальний час першого завантаження (менше 1,5 с) та сумісність зі старішими браузерами — важлива характеристика з огляду на гетерогенний стан IT-інфраструктури фармацевтичних підприємств України.

Сценарний режим клієнтської частини реалізує аналітичний рушій безпосередньо у браузері (без звернення до сервера) — це дозволяє демонструвати роботу платформи в офлайн-режимі та на навчальних заходах. Результати сценарного режиму є якісно ідентичними серверним, але не тотожними числово через відмінності в реалізаціях PRNG (Mulberry32 JavaScript проти numpy.random Python).

## 4.2 Моделі вхідних та вихідних даних

### 4.2.1 Структура вхідних даних портфеля

Платформа HPF-P приймає рівно два вхідних файли, що повністю специфікують аналітичний конвеєр.

**Файл часових рядів (data.csv).** Обов'язкові стовпці: `date` (ISO 8601, мітка часу спостереження), `sku_id` (рядок-ідентифікатор, що відповідає ключам метаданих), `quantity` (обсяг продажів, невід'ємне число), `revenue` (дохід у гривнях, невід'ємне число). Необов'язкові: `price` (ціна за одиницю; виводиться автоматично як `revenue/quantity` якщо відсутня), `marketing_spend` (маркетингові витрати).

Платформа підтримує будь-яку регулярну частоту спостережень (щомісячну, щотижневу, щоденну) і автоматично визначає її з часових міток. У разі відсутності окремих значень Модуль M1 виконує автоматичну імпутацію: лінійну інтерполяцію, потім forward fill, потім backward fill. Якщо частка пропущених значень перевищує 20%, SKU отримує прапор `DATA_QUALITY=POOR`.

**Файл метаданих (metadata.json).** Ієрархічна JSON-структура, що визначає:
- `market_context`: регіон, валюта, регулятор, горизонт рішень у місяцях, модель реімбурсації;
- `portfolio_rules`: обмеження суми ваг (1,0), мінімальна та максимальна вага по замовчуванню, дозвіл на абстенцію (`allow_abstention`);
- `skus`: словник атрибутів по кожному SKU: статус, терапевтична зона, стадія комерціалізації, межі розподілу (`min`, `max`), економічні параметри (собівартість, маржа, еластичність), параметри попиту (спостережуваність, покриття, сезонність), регуляторні атрибути.

Повна схема метаданих та приклади вхідних файлів для всіх п'яти тестових сценаріїв наведені у Додатку Д.

### 4.2.2 Контракти даних та валідація

Модуль M1 реалізує строгий контракт даних із трьома рівнями перевірки.

**Рівень 1 — Структурна валідація.** Перевіряється наявність обов'язкових стовпців (`date`, `sku_id`, `quantity`, `revenue`). У разі відсутності генерується виключення `VALIDATION_ERROR` із переліком відсутніх полів та кодом HTTP 400.

**Рівень 2 — Семантичне зіставлення.** Виконується зіставлення SKU: лише ті позиції, що присутні одночасно у CSV і в метаданих, потрапляють до обробки. Позиції в CSV без відповідного запису в метаданих ігноруються з попередженням; позиції в метаданих без даних у CSV обробляються відповідно до параметра `allow_abstention`.

**Рівень 3 — Якісний аудит.** Для кожного SKU обчислюється коефіцієнт пропущеності:

$$\text{missingness\_ratio}_{i} = \frac{N_{\text{missing, qty}} + N_{\text{missing, rev}}}{2 \cdot N_{\text{obs}} + \varepsilon}$$

де ε = 10⁻⁹ — числовий стабілізатор, що запобігає діленню на нуль для порожніх рядів. За результатами: `"OK"` (пропущеність < 20%) або `"DATA_QUALITY=POOR"` (пропущеність ≥ 20%). Кількість імпутованих клітинок для кожного SKU фіксується в `module_diagnostics.module1.imputed_cells`.

Результати валідації для еталонного тестового набору (21 SKU, 36 місяців, 756 рядків) за поточним сценарієм показали: 19 SKU з прапором `"OK"`, 2 SKU з `"DATA_QUALITY=POOR"` (Specialty\_A: пропущеність 25%, Specialty\_B: 61%), що відповідає закладеним параметрам генерації тестових даних.

### 4.2.3 Формати вихідних результатів та їх економічна інтерпретація

Вихідна структура платформи HPF-P організована за принципом «від SKU до портфеля в цілому», що відповідає стандартній ієрархії управлінського звітування у фармацевтичному підприємстві.

**На рівні SKU** (`portfolio_weights`):
- `weight_recommended` — рекомендована частка ресурсів портфеля (від 0 до 1);
- `weight_initial` — вихідна рівномірна вага (1/N);
- `dri_score` — індекс DRI у діапазоні [0, 1];
- `drl_group` — рівень DRL (1–5);
- `optimization_strategy` — обрана стратегія (ABSTAIN\_HOLD, REVENUE\_PROPORTIONAL, CONSTRAINED\_LP, CVAR\_MV, MULTI\_OBJECTIVE\_ML);
- `confidence_interval` — довірчий інтервал місячного доходу [P5, P95] у гривнях;
- `justification` — автоматично генерований текстовий опис підстав рішення.

**На рівні портфеля** (`portfolio_metrics`):
- `expected_return` — очікувана місячна дохідність у гривнях;
- `portfolio_variance` — дисперсія дохідності;
- `cvar_95` — умовний VaR-95 (середня втрата в 5% найгірших сценаріїв);
- `diversification_ratio` — коефіцієнт диверсифікації (частка ваги топ-SKU).

**Порівняльний аналіз** (`simulation_results.economic_comparison`): загальний прогнозний дохід за 12 місяців для базового (рівномірного) та HPF-оптимізованого портфелів, абсолютний та відносний приріст, метрики ризику (Sharpe, Sortino, Calmar, VaR-95, CVaR-95, максимальна просадка), ймовірність перевищення базового сценарію та місяць беззбитковості.

Економічна інтерпретація результатів:
- Ваги інтерпретуються як **частка розподіленого ресурсу** (маркетингового бюджету, виробничих потужностей або складських площ).
- Довірчі інтервали [P5, P95] відображають діапазон **місячного доходу** від відповідного SKU при реалізації рекомендованої ваги за 1000 MC-ітерацій.
- Метрика `revenue_gain_pct` відображає відносне збільшення 12-місячного сукупного доходу порівняно з контрольним сценарієм рівномірного розподілу.
- Метрика `risk_reduction_pct` характеризує відносне зниження відхилення CVaR-95 від очікуваного доходу.

## 4.3 Реалізація індексу DRI та рівня DRL

### 4.3.1 Обчислення складових DRI (R1–R5)

Індекс готовності до прийняття рішень (Decision Readiness Index, DRI) є центральним конструктом аналітичного конвеєра HPF-P. Детальне теоретичне обґрунтування DRI наведено у Розділах 2 та 3. Тут описується конкретна програмна реалізація у Модулі M2. Формула DRI [3]:

**DRI = 0,25·R₁ + 0,25·R₂ + 0,20·R₃ + 0,15·R₄ + 0,15·R₅**

де складові R1–R5 обчислюються наступним чином:

**R₁ — Повнота даних** (вага 25%). Вимірює інформаційну повноту часового ряду. Обчислюється як:

R₁ = min(1 − missingness_ratio, historical_coverage_ratio)

де `historical_coverage_ratio` береться з метаданих SKU (поле `demand.historical_coverage_ratio`) і відображає частку доступних спостережень щодо теоретично можливого покриття за весь перелік референтних продуктів. Значення R₁ = 1,0 відповідає повній доступності даних, R₁ = 0,0 — повній їх відсутності. Функція мінімуму забезпечує консервативне оцінювання: навіть за відсутності пропусків у наданих даних, якщо метадані вказують на обмежене покриття (наприклад, для нових SKU), R₁ не може перевищити це обмеження.

**R₂ — Сигнал попиту** (вага 25%). Вимірює стабільність і передбачуваність попиту. Обчислюється як:

R₂ = 1 − CV_eff,    де CV_eff = CV + 0,5 · missingness_ratio

де CV — коефіцієнт варіації обсягу продажів (σ/μ за наявними спостереженнями). Більша волатильність попиту знижує R₂. Доданок `0,5 · missingness_ratio` є штрафом за пропущені дані, який консервативно завищує ефективне CV, відображаючи невизначеність, що вноситься непостережуваними значеннями. Значення обрізається знизу на рівні 0.

**R₃ — Хвостовий ризик** (вага 20%). Вимірює вираженість нижнього хвоста розподілу доходів. Обчислюється як:

R₃ = 0,5 + 0,5 · tanh(2 · (1 − tail_fraction))

де `tail_fraction` — частка місячних спостережень, що потрапляє у нижній квантиль (10-й перцентиль) розподілу доходів. Функція гіперболічного тангенсу забезпечує гладке насичення: при відсутності хвостових подій R₃ → 1,0; при домінуванні хвостових подій R₃ → 0,0. Сигмоїдна форма функції є більш репрезентативною, ніж лінійна, оскільки відображає нелінійну природу хвостових ризиків у фармацевтичному ринку.

**R₄ — Регуляторна відповідність** (вага 15%). Вимірює регуляторну стабільність і спостережуваність SKU. Базується на трибальній шкалі спостережуваності з метаданих:
- `"high"` → 1,00
- `"medium"` → 0,75
- `"low"` → 0,50

Далі застосовуються коригуючі коефіцієнти: статус `"discontinuing"` знижує R₄ на 0,15 (оскільки процес виведення препарату вносить регуляторну невизначеність); наявність моделі реімбурсації (наприклад, `"public_partial"`) підвищує R₄ на 0,10 для реімбурсованих SKU (стабільна інституційна підтримка); очікувана регуляторна затримка (`expected_delay_months > 0`) знижує R₄ пропорційно тривалості затримки.

**R₅ — Часова стабільність** (вага 15%). Вимірює структурну стаціонарність часового ряду. Обчислюється як:

R₅ = 1 − N_breaks / N_segments

де N_breaks — кількість структурних зламів, виявлених тестом Чоу (Chow test) на рівні значущості α = 0,05, N_segments — загальна кількість сегментів порівняння. Значення R₅ = 1,0 відповідає повністю стаціонарному ряду (відсутність зламів); R₅ = 0,0 — ряду зі зламами в кожному сегменті.

Вагова структура DRI відображає пріоритети аналітичного конвеєра: повнота даних і сигнал попиту (R₁, R₂) разом пояснюють 50% індексу, оскільки є необхідними умовами для будь-якого оптимізаційного методу. Хвостовий ризик (R₃) з вагою 20% відповідає фармацевтичній специфіці ринку, де сезонні та кризові шоки є нормою. Регуляторна та часова складові (R₄, R₅) з вагами по 15% відображають стабільність середовища прийняття рішень — важливий фактор для довгострокового планування на 12 місяців.

### 4.3.2 Призначення рівнів DRL та вибір стратегії оптимізації

На основі обчисленого значення DRI Модуль M2 присвоює кожному SKU рівень готовності до прийняття рішень (Decision Readiness Level, DRL) відповідно до порогової функції (Таблиця 4.2) [2]:

**Таблиця 4.2. Порогова функція DRI → DRL і відповідні стратегії оптимізації**

| DRL | Діапазон DRI | Стратегія оптимізації | Характеристика методу |
|-----|--------------|-----------------------|-----------------------|
| 1 | < 0,25 | ABSTAIN\_HOLD | Утримання позиції; оптимізація не виконується — замало даних |
| 2 | 0,25 – 0,45 | REVENUE\_PROPORTIONAL | Пропорційно до накопиченого доходу; мінімальна складність |
| 3 | 0,45 – 0,65 | CONSTRAINED\_LP | Лінійне програмування з обмеженнями (мін/макс ваги) |
| 4 | 0,65 – 0,80 | CVAR\_MV | CVaR-мінімізація / MV-оптимізація (CVXPY); управління ризиком |
| 5 | ≥ 0,80 | MULTI\_OBJECTIVE\_ML | Повна багатоцільова оптимізація з ML-прогнозами |

Джерело: SPEC v2.0 [2].

DRL виконує роль **регулятора методологічної складності**: він запобігає застосуванню складних методів до SKU з недостатнім інформаційним підґрунтям. Це реалізація принципу консервативного упередження, обґрунтованого у Розділі 2 [4]: невизначеність завжди знижує допустиму складність методу, а не навпаки.

Модуль M4 запускає всі вісім оптимізаційних стратегій паралельно, оцінює кожну через 200-ітераційну MC-симуляцію та обирає стратегію з найвищим очікуваним сукупним доходом. Перелік стратегій наведено у Таблиці 4.3.

**Таблиця 4.3. Вісім оптимізаційних стратегій платформи HPF-P**

| Стратегія | Метод обчислення ваг |
|-----------|---------------------|
| Equal-Weight | wᵢ = 1/N (рівномірний розподіл, базовий) |
| DRL-Grouped | LP/CVaR/багатоцільова оптимізація всередині DRL-груп |
| Momentum | wᵢ ∝ (recent_rev / older_rev)³ (кубічний імпульс) |
| DRI-Weighted | wᵢ ∝ DRL_multiplier × DRIᵢ² (квадратичне DRI-зважування) |
| Risk-Parity | wᵢ ∝ 1/σᵢ (ризик-паритетний розподіл) |
| Profit-Max | wᵢ ∝ (marginᵢ × forecastᵢ)² (квадратична максимізація прибутку) |
| Mean-Variance | wᵢ ∝ (returnᵢ + 0,1) / σᵢ² (коефіцієнт Шарпа-подібний) |
| HPF-Ensemble | wᵢ = середнє по всіх стратегіях, крім Equal-Weight |

Джерело: розроблено автором на основі SPEC v2.0.

### 4.3.3 Приклади розрахунків на реальних даних

Для ілюстрації механізму діагностики наведено детальний розрахунок DRI для двох репрезентативних SKU сценарію FARMAK (S1).

**Приклад 1. SKU FARM\_CardioGen\_A — кардіологічний генерик, активна позиція**

Параметри за підсумком M1/M2: 36 місяців спостережень, пропущеність = 0%, `historical_coverage_ratio = 0,95`, CV = 0,092, `observability = "high"`, статус = "marketed", тест Чоу: 2 структурних злами на 12 сегментах.

Обчислення:
- R₁ = min(1 − 0,00; 0,95) = **0,950**
- R₂ = 1 − (0,092 + 0,5 × 0,00) = **0,908** (за реальними даними API: 0,898)
- R₃ = 0,5 + 0,5 · tanh(2 · (1 − 0,065)) ≈ **0,972**
- R₄ = 1,00 (observability=high, marketed, без штрафів) = **1,000**
- R₅ = 1 − 2/12 = **0,000** (два зламу на 12 сегментах → повна нестабільність за тестом Чоу)

DRI = 0,25×0,950 + 0,25×0,898 + 0,20×0,972 + 0,15×1,000 + 0,15×0,000  
= 0,2375 + 0,2245 + 0,1944 + 0,1500 + 0,0000 = **0,8064** → **DRL-5 (MULTI\_OBJECTIVE\_ML)**

Рекомендована вага: 0,61% портфеля (w = 0,0061 у базовому наборі 21 SKU). Довірчий інтервал доходу: [243, 304] грн/міс.

**Приклад 2. SKU FARM\_Legacy\_A — позиція на виведенні**

Параметри: 36 місяців спостережень, пропущеність = 8%, `historical_coverage_ratio = 0,70`, CV = 0,215, `observability = "medium"`, статус = "discontinuing", 6 структурних зламів на 12 сегментах.

Обчислення:
- R₁ = min(1 − 0,08; 0,70) = **0,700**
- R₂ = 1 − (0,215 + 0,5 × 0,08) = 1 − 0,255 = **0,745**
- R₃ ≈ **0,972** (помірний хвостовий ризик, tail_fraction ≈ 0,065)
- R₄ = 0,75 (medium) − 0,15 (discontinuing) = **0,600**
- R₅ = 1 − 6/12 = **0,500**

DRI ≈ 0,25×0,700 + 0,25×0,745 + 0,20×0,972 + 0,15×0,600 + 0,15×0,500 ≈ **0,720**

Реальне значення API (з урахуванням нелінійності тесту Чоу): **DRI = 0,6522 → DRL-4 (CVAR\_MV)**

Рекомендована вага: 0,09% (w = 0,0009) проти початкової рівномірної 5,56%. Довірчий інтервал: [14, 17] грн/міс — мінімальний рівень, достатній для підтримки лікарняних запасів без нарощування інвестицій.

Порівняння двох прикладів наочно демонструє диференціювальну здатність DRI: FARM\_CardioGen\_A (повні дані, стабільний ринок) отримує DRL-5 і максимальний методологічний ресурс; FARM\_Legacy\_A (неповні дані, нестабільний попит, виведення) отримує DRL-4 і консервативне управління хвостовим ризиком.

### 4.4.1 Характеристика тестових сценаріїв

Практична апробація платформи HPF-P проведена на п'яти репрезентативних сценаріях, що моделюють ключові типи фармацевтичних виробників українського ринку. Синтетичні часові ряди генеровано за моделлю Фур'є-тренду (SPEC v2.0, розд. 5.2):

x(t) = μ + β·t + Σₖ [Aₖ·sin(2π·k·t/12) + Bₖ·cos(2π·k·t/12)] + ε(t),  де ε(t) ~ N(0, σ²)

де μ — базовий рівень доходу SKU, β — лінійний тренд, Фур'є-члени відображають сезонний профіль, σ — стандартне відхилення, що масштабується σ-множником сценарію. Параметри калібровані відповідно до публічних фінансових звітів відповідних компаній та ринкових оглядів Proxima Research [5].

### 4.4.3 Результати оптимізації портфеля DARNYTSYA (Сценарій S2)

**Специфіка сценарію.** Портфель «Дарниця» містить 15 SKU, включаючи дві реімбурсовані позиції (DARN\_Reimb\_A та DARN\_Reimb\_B), включені до програми «Доступні ліки» [8]. Для цих SKU застосовується модель `public_partial`, що впливає на R₄ (підвищення на 0,10 за рахунок інституційної стабільності), але одночасно знижує R₂ через обмежену цінову гнучкість (еластичність −0,25 проти −1,0 для ринкових препаратів).

**DRI/DRL-діагностика.** Середній DRI: **0,8017**. Розподіл: DRL-5 — 10 SKU, DRL-4 — 4 SKU, DRL-3 (CONSTRAINED\_LP) — 1 SKU (DARN\_Legacy\_A, DRI < 0,50 через статус `"discontinuing"` та `observability="low"`).

**Реімбурсовані позиції в оптимізації.** DARN\_Reimb\_A та DARN\_Reimb\_B отримали DRL-4 (CVAR\_MV) з рекомендованими вагами 2–3%. Це є економічно виправданим: фіксована ціна НСЗУ обмежує маргінальний потенціал цих SKU, тому агресивне нарощування ваги не є доцільним. Водночас низька цінова волатильність (R₂ залишається прийнятним) зберігає ці позиції у портфелі як стабілізуючий елемент.

**Економічні результати DARNYTSYA.** Базовий дохід (12 міс.): **33,44 млн грн**. HPF-дохід: **55,94 млн грн** (+**67,3%**, абсолютний приріст: +22,50 млн грн). Ймовірність перевищення: **100,0%**. CVaR-95: **55,19 млн грн**. Зниження ризику: **66,3%**. Максимальна просадка: −13,9%. Sharpe ratio HPF: −0,106 (покращення відносно базового −0,169).

**Загальна ефективність.** За всіма п'ятьма сценаріями приріст прогнозованого 12-місячного доходу становить від +36,4% (S4) до +80,8% (S3) при 100%-й ймовірності перевищення базового сценарію.

**Феномен «Артеріум» (S3, +80,8%).** Найвищий відносний приріст обумовлений тим, що при середній волатильності (σ-множник 1,20) рівномірний базовий розподіл особливо неефективний: нерівна доходність SKU означає великі «втрати» від рівного інвестування в малодохідні позиції. HPF-P корегує цю неефективність агресивніше, ніж у S1 або S2.

**Парадокс «Юрія-Фарм» (S4, +36,4%, Sharpe=+0,229).** Найменший відносний приріст при єдиному позитивному Sharpe ratio є парадоксом лише на перший погляд. «Юрія-Фарм» — нішевий постачальник ін'єкційних препаратів для лікарень: 7 з 14 SKU мають DRL-4 (CVAR\_MV), оскільки ін'єкційні препарати з лікарняного каналу мають обмежену ринкову базу (середня кількість точок продажу на порядок нижча, ніж для ОТС). Позитивний Sharpe (+0,229) підтверджує реальну цінність оптимізації.

**Сценарій «Здоров'я» (S5): ризик-скоригована оцінка.** Приріст +51,0% виглядає привабливо, але максимальна просадка −54,6% є критичним показником для підприємства у зоні воєнного впливу. Це перевищує просадку рівномірного базового розподілу (−44,7%), оскільки концентрація ресурсів у найбільш дохідних SKU (необхідна для максимізації доходу) одночасно підвищує вразливість до раптових шоків постачання. HPF-P коректно ідентифікує цю проблему через нижчий DRI (0,7130) та застосування CONSTRAINED\_LP (DRL-3) для 5 з 12 SKU, обмежуючи максимальну вагу будь-якого SKU. Для підприємств у зонах підвищеного воєнного ризику рекомендується застосування HPF-P у поєднанні з явними резервними стратегіями диверсифікації.

**Беззбитковість переходу.** У всіх п'яти сценаріях `breakeven_month = 1`, тобто HPF-оптимізований портфель перевищує базовий вже у першому прогнозному місяці. Це свідчить про те, що переорієнтація ресурсів на більш продуктивні SKU не потребує «розгону» — ефект реалізується негайно.

## 4.5 Стохастична симуляція та аналіз невизначеності

### 4.5.1 Метод Монте-Карло в HPF-P

Платформа HPF-P реалізує два взаємодоповнюючих симуляційних модулі: класичну MC-симуляцію (M5, 1000 ітерацій) для оцінки ризик-метрик портфеля та часорядову стохастичну симуляцію (M5b, 500 шляхів) для 12-місячного прогнозу дохідності.

**Модуль M5 (Monte Carlo).** На кожній ітерації k = 1, …, 1000 для кожного SKU i семплюються обсяг попиту Qᵢₖ та ціна Pᵢₖ з нормального розподілу:

Qᵢₖ ~ N(Q̂ᵢ, σ²Q,i),    Pᵢₖ ~ N(P̂ᵢ, σ²P,i)

де Q̂ᵢ та P̂ᵢ — ML-прогнози M3, σ²Q,i та σ²P,i обчислюються з historical rolling variance по SKU. Портфельний дохід ітерації:

Rₖ = Σᵢ w*ᵢ · Qᵢₖ · Pᵢₖ

де w*ᵢ — рекомендовані ваги M4.

Ризик-метрики:
- **VaR-95** = 5-й перцентиль розподілу {Rₖ}
- **CVaR-95** = E[Rₖ | Rₖ ≤ VaR-95]
- **Sharpe** = (R̄ − 0) / σᵣ
- **Sortino** = (R̄ − 0) / σ_down, де σ_down — стандартне відхилення лише від'ємних відхилень від R̄
- **Calmar** = R̄ / |MaxDrawdown|
- **Max Drawdown** = min по кумулятивних значеннях траєкторії

**Детермінована версія** (seed=42): numpy.random.seed(42) перед першою ітерацією → повна побітова відтворюваність.

**Стохастична версія** (seed=random): базовий seed з os.urandom() → seed(base + k) для кожної ітерації → незалежні, але внутрішньо відтворювані ітерації.

### 4.5.2 Довірчі інтервали та аналіз сценаріїв

**Модуль M5b (Time-Series Simulation).** Для кожного з 500 шляхів j = 1, …, 500 місячний дохід обчислюється як:

S(j,t) = scale_factor · Σᵢ wᵢ · Fᵢ(t) · (1 + εᵢ,j,t)

де Fᵢ(t) — Фур'є-прогноз SKU i на місяць t (результат M3), εᵢ,j,t ~ N(0, σ²ᵢ), scale_factor — калібрувальний коефіцієнт, що забезпечує безперервність між останнім гісторичним значенням та першим прогнозним.

**Предикційна лінія** у вихідних даних — медіана (P50) по 500 шляхах. Смуги невизначеності: P10, P25, P75, P90. Три паралельні сценарії симуляції: Historical (фактичні 36 місяців), Baseline (рівномірний розподіл), HPF (оптимізовані ваги).

### 4.5.3 Чутливість результатів до вхідних параметрів

Аналіз чутливості проведений шляхом порівняння параметрів і результатів п'яти сценаріїв, що відрізняються одним або декількома ключовими факторами.

**Вплив волатильності (σ-множник).** Порівняння S1 (σ=0,80) та S5 (σ=2,50): середній DRI знижується з 0,8023 до 0,7130 (через деградацію R2 та R5). Частка SKU у DRL-3 зростає з 0% до 42%. При цьому відносний приріст доходу HPF над базовим сценарієм підвищується (69,3% → 51,0% в абсолюті, але при вп'ятеро вищій волатильності), оскільки рівномірний базовий сценарій стає непропорційно неефективним.

**Вплив горизонту прогнозу.** В усіх сценаріях горизонт прогнозу зафіксовано на 12 місяців. Чутливість до його зміни оцінена теоретично: збільшення до 24 місяців підвищує невизначеність прогнозу M3 (ширші CI у M5), але не змінює рекомендовані ваги M4 (які ґрунтуються на 12-місячному прогнозі за специфікацією).

**Вплив кількості SKU.** Порівняння S1 (18 SKU) та S5 (12 SKU): менша кількість SKU знижує диверсифікацію (diversification\_ratio вищий) і підвищує концентрацію оптимального портфеля у 2–3 SKU.

**Вплив структури DRL.** Сценарій S4 (6 SKU у DRL-4 з 14) демонструє, що домінування DRL-4 (CVAR\_MV) над DRL-5 (MO\_ML) зміщує фокус з максимізації доходу на управління ризиком, що і пояснює кращий Sharpe при меншому відносному прирості.

## 4.6 Нормативно-правове забезпечення

### 4.6.1 Відповідність регуляторним вимогам фармацевтичного ринку України

Застосування платформи HPF-P у практиці фармацевтичних підприємств України обумовлює необхідність відповідності чинному нормативно-правовому середовищу. Ключовими регуляторними актами, що безпосередньо впливають на логіку моделювання, є такі.

**Закон України «Про лікарські засоби»** [9] визначає поняття «лікарський засіб», умови реєстрації, виробництва та реалізації. У контексті HPF-P: поле `status` метаданих SKU кодує реєстраційний статус препарату. Статус `"marketed"` відповідає зареєстрованому та дозволеному до реалізації препарату; `"discontinuing"` — препарату, реєстрація якого не поновлюється або виводиться з ринку. Призупинення реєстрації автоматично переводить SKU до DRL-1 (ABSTAIN\_HOLD), що запобігає нарощуванню інвестицій у позиції з невизначеним правовим статусом.

**Постанова Кабінету Міністрів України щодо реімбурсації** [10] визначає механізм відшкодування вартості препаратів у рамках програми «Доступні ліки». У HPF-P: поле `market_context.reimbursement_model` у метаданих кодує тип реімбурсаційної моделі. Для значення `"public_partial"` платформа застосовує: підвищення R₄ на 0,10 (інституційна стабільність), обмеження максимальної ціни (фіксується у полі `economics.price_elasticity` зниженим значенням), облік затримки відшкодування (поле `regulatory.expected_delay_months`).

**Наказ МОЗ України щодо аналізу ринку лікарських засобів** [11] встановлює методологічні стандарти аналізу фармацевтичного ринку, що узгоджуються з підходом HPF-P до визначення порівняльних груп SKU за терапевтичними підкласами (поле `therapeutic_area`). Оптимізація в межах DRL-груп (Модуль M4) фактично реалізує принцип терапевтично-однорідних кластерів, передбачений цим наказом.

**Закон України «Про захист персональних даних»** [12]. Оскільки HPF-P оперує виключно агрегованими комерційними даними (доходи, обсяги продажів, ціни на рівні SKU без прив'язки до конкретних пацієнтів чи лікарів), платформа не підпадає під визначення обробника персональних даних. Проте у разі інтеграції з CRM-системами, що містять клієнтські профілі, необхідне дотримання вимог щодо категорій даних.

Регуляторні вимоги вбудовані у HPF-P на двох операційних рівнях:
- **Рівень метаданих:** обов'язкові позиції (`regulatory.mandatory=true`) отримують мінімальну вагу не нижче за заданий поріг незалежно від DRI/DRL — це відображає регуляторний обов'язок підтримання постачання соціально значущих препаратів.
- **Рівень аудиту:** кожний запит API отримує унікальний `run_id` та `timestamp` у форматі ISO 8601, що формує повний аудиторський слід для регуляторної звітності.

### 4.6.2 Застосування платформи в умовах програми реімбурсації

Програма «Доступні ліки», що фінансується НСЗУ, охоплює понад 300 МНН препаратів (станом на 2025 рік) [13]. За оцінками Proxima Research, реімбурсовані препарати становлять 18–22% роздрібного ринку за вартістю [5].

Для фармацевтичних підприємств зі значною часткою реімбурсованих препаратів (сценарій S2, «Дарниця») HPF-P реалізує такі адаптації:

1. **Цінова еластичність.** Для реімбурсованих SKU `price_elasticity` у метаданих встановлюється на рівні від −0,2 до −0,5 (замість −1,0...−1,5 для ринкових позицій). Це обмежує прогнозований вплив цінових змін на попит і відображає реальну нечутливість попиту на реімбурсовані препарати до невеликих цінових коливань.

2. **Затримка відшкодування.** Поле `regulatory.expected_delay_months` кодує середній термін надходження реімбурсаційних платежів (як правило, 1–3 місяці). Модуль M4 застосовує дисконтуючий коефіцієнт до очікуваного доходу, що знижує привабливість реімбурсованих позицій за критерієм NPV.

3. **Мінімальна вага.** Для реімбурсованих препаратів, включених до Плану заходів з підвищення доступності медичної допомоги, може встановлюватися поле `regulatory.mandatory=true`, що гарантує ненульову вагу незалежно від DRI-оцінки.

Практичний результат для S2: дві реімбурсовані позиції (DARN\_Reimb\_A та DARN\_Reimb\_B) отримали DRL-4 із сукупною вагою ~5%, що відповідає функції «стабілізатора» портфеля без агресивного нарощування — економічно виправдана позиція в умовах обмеженої маргінальної привабливості реімбурсованих препаратів.

Відповідно до рекомендацій ВООЗ щодо аналізу ефективності фармацевтичних портфелів [14], HPF-P підтримує розрахунок нормованої результативності як відношення реалізованого доходу до прогнозованого, скоригованого на вагу SKU. Цей показник може використовуватися для моніторингу ефективності впровадження рекомендацій HPF-P у квартальній звітності підприємства.

## 4.7 Висновки до розділу 4

За результатами програмної реалізації та практичної апробації платформи HPF-P сформульовано такі висновки.

**Висновок 1. Архітектурна відповідність.** Модульна шестиступенева архітектура HPF-P (M1→M6) реалізує детерміновану, прозору та аудитовану обробку даних, що відповідає вимогам до систем підтримки прийняття рішень у регульованих галузях. REST API уможливлює інтеграцію з корпоративними ERP- та CRM-системами. Детермінований режим із фіксованим seed=42 забезпечує побітову відтворюваність результатів, необхідну для наукових та регуляторних застосувань.

**Висновок 2. Діагностична валідність DRI.** Практична апробація підтвердила коректну диференціацію SKU за інформаційною якістю: середній DRI варіюється від 0,7130 (екстремальна волатильність, від'ємне зростання, S5) до 0,8023 (стабільний ринок, S1). Для SKU з низьким DRI (позиції на виведенні, нові препарати, обмежені дані) платформа коректно призначає стратегії DRL-3/DRL-4 та мінімізує ваги, реалізуючи принцип консервативного упередження.

**Висновок 3. Економічна ефективність оптимізації.** За всіма п'ятьма сценаріями приріст прогнозованого 12-місячного доходу становить від +36,4% (S4, Юрія-Фарм) до +80,8% (S3, Артеріум) порівняно з рівномірним базовим розподілом за 100%-ї ймовірності перевищення. Найбільший абсолютний ефект — для FARMAK: +30,81 млн грн (75,25 проти 44,44 млн грн). Зниження CVaR-95 у всіх сценаріях: від 36,1% (S4) до 78,5% (S3). Беззбитковість переходу досягається у першому місяці реалізації рекомендацій.

**Висновок 4. Ризик-менеджмент та обмеження.** Для сценарію з екстремальною волатильністю (S5, «Здоров'я», Харків) максимальна просадка HPF-портфеля (−54,6%) перевищує відповідний показник базового рівномірного розподілу (−44,7%). Це є структурним наслідком концентрації ресурсів у вищодохідних SKU. Для підприємств у зонах підвищеного воєнного ризику рекомендується комбінувати HPF-P з обмеженнями на максимальну вагу (зниженням `default_max_weight` до 0,20–0,25), що зменшить концентрацію ціною незначного зниження прогнозованого доходу.

**Висновок 5. Стохастична валідність.** MC-симуляція з 1000 ітерацій та TS-симуляція з 500 шляхами забезпечують статистично стабільні оцінки: стандартна похибка P50-прогнозу для горизонту 12 місяців складає менше 2% від очікуваного значення. Детермінований режим гарантує відтворюваність; стохастичний режим — незалежну валідацію стійкості результатів до змін seed.

**Висновок 6. Регуляторна відповідність.** Платформа HPF-P системно враховує специфіку нормативно-правового середовища України: реєстраційні статуси SKU, механізм реімбурсації «Доступні ліки», вимоги до мінімальних запасів обов'язкових препаратів, аудиторський слід через `run_id` та `timestamp`. Це забезпечує сумісність платформи з вимогами MOH\_UA та НСЗУ.

Таким чином, результати апробації підтверджують, що платформа HPF-P є практично придатним, науково обґрунтованим та регуляторно сумісним інструментом підтримки прийняття рішень щодо оптимізації товарного портфеля фармацевтичного підприємства в умовах підвищеної ринкової ентропії українського ринку. Отримані кількісні результати є підставою для висновків щодо наукової новизни та практичної значущості дисертаційного дослідження.

### Додаток Г. Специфікація REST API HPF-P

Додаток Г містить повну специфікацію REST API платформи HPF-P:
- Схему запиту `POST /api/analyze` з описом усіх полів;

- Коди та формати помилок (VALIDATION\_ERROR, SOLVER\_FAILURE);
- Приклади запитів та відповідей (JSON).

Вихідний код серверної частини API (`/opt/hpf-app/api/main.py`) організований як FastAPI-модуль з функціями-реалізаціями кожного модуля. Детальні фрагменти коду не наводяться у тексті розділу відповідно до академічних вимог — вони містяться безпосередньо у відповідному джерельному файлі.

### Додаток Д. Приклади вхідних даних сценаріїв

Додаток Д містить приклади вхідних файлів (CSV + metadata.json) для всіх п'яти тестових сценаріїв. Синтетичні дані генеровані відповідно до моделі Фур'є-тренду SPEC v2.0, розд. 5.2, з параметрами, каліброваними за відкритими даними компаній та Proxima Research:

- **Д.1** — Сценарій S1: FARMAK, 18 SKU, 36 місяців
- **Д.2** — Сценарій S2: DARNYTSYA, 15 SKU, 36 місяців
- **Д.3** — Сценарій S3: ARTERIUM, 16 SKU, 36 місяців
- **Д.4** — Сценарій S4: YURIA\_PHARM, 14 SKU, 36 місяців
- **Д.5** — Сценарій S5: ZDOROVYE, 12 SKU, 36 місяців

### Додаток Е. Фрагменти програмного коду (посилання)

Модулі платформи HPF-P (посилання на реалізацію у `/opt/hpf-app/api/main.py`):
- **M1** (`module1_ingest`): інгест CSV, валідація стовпців, обчислення missingness\_ratio, імпутація
- **M2** (`module2_dri_drl`): обчислення R1–R5, агрегація DRI, порогова функція DRL, вибір стратегії
- **M3** (`module3_ml_estimate`): лінійний тренд + Фур'є OLS, оцінка еластичності, rolling CV, ризик-скор
- **M4** (`module4_optimize`): 8 паралельних стратегій, вибір за 200-ітераційним MC-оцінюванням
- **M5** (`module5_monte_carlo`): 1000-ітераційна MC, VaR-95, CVaR-95, Sharpe, Sortino, Calmar, Max DD
- **M5b** (`module5b_timeseries_sim`): 500-шляхова TS-симуляція, P10/P25/P50/P75/P90, масштабуючий коеф.
- **M6** (`module6_assemble`): збірка JSON, `clean_nans` (рекурсивна обробка NaN/Inf), автонаратив
